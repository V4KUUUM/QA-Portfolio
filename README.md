# Тестирование API сервиса «Яндекс.Прилавок»

## 📌 Описание задачи
Разработчики передали на тестирование новую функциональность API. Нужно было проверить работу с наборами (kits), корзиной (orders) и расчет доставки курьерской службой.

**Основные ручки:**
* `POST /api/v1/kits/{id}/products` — добавление продуктов в набор.
* `POST /fast-delivery/v3.1.1/calculate-delivery.xml` — расчет стоимости доставки (XML).
* `GET /api/v1/orders/:id` — получение состава корзины.
* `PUT /api/v1/orders/:id` — добавление в корзину.

## ⚙️ Предусловия
* **Документация:** Swagger/Apidoc.
* **Окружение:** Тестовый стенд Яндекс Практикума.
* **Инструменты:** Postman, cURL.

## 🛠 Что было сделано
1. Спроектирован чек-лист на **110 проверок** (позитивные и негативные сценарии, граничные значения, спецсимволы, SQL-инъекции).
2. Проведено ручное тестирование REST-ручек с передачей данных в форматах JSON и XML.
3. Заведено **20 баг-репортов** (включая критические ошибки валидации).

## 🐛 Главные найденные баги

Я выделил самые критичные ошибки, которые приводят к падению сервера или нарушению логики:

### [Б19] Ошибка 500 при неверном типе данных (Input Validation)
* **Суть:** Если отправить объект вместо массива в `productsList`, бэкенд падает, так как не умеет обрабатывать этот тип данных.
* **Запрос:** `curl -X PUT ... -d '{ "productsList": { "id": 7 } }'`
* **Результат:** `500 Internal Server Error` (вместо 400 Bad Request).

### [Б1] Добавление несуществующего продукта
* **Суть:** Система позволяет добавлять в наборы товары с ID, которых нет в базе данных.
* **Результат:** `200 OK` (Нарушение целостности данных).

### [Б4] Ошибка 500 при переполнении числа (Integer Overflow)
* **Суть:** При передаче слишком большого числа в `quantity` (например, 9999999999) сервер отдает 500-ю ошибку с текстом из БД, раскрывая детали реализации.
* **Результат:** `500 Internal Server Error`.
